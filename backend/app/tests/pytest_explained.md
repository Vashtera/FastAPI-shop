# Разбор тестов: test_register и test_login

Этот документ подробно объясняет каждую строку двух тестов — регистрации и логина.
Цель — понять не просто "что писать", а **почему именно так**.

---

## Импорт

```python
from httpx import AsyncClient
```

Импортируем класс `AsyncClient` только ради **типизации** (type hint). Он не создаёт клиент сам —
клиент придёт через фикстуру `client` из `conftest.py`. Этот импорт нужен только чтобы редактор
(VS Code / Pylance) знал: "переменная `client` — это объект `AsyncClient`, у него есть методы
`.get()`, `.post()` и т.д." — и подсказывал их при наборе кода.

---

## Тест 1 — Регистрация

```python
async def test_register(client: AsyncClient):
```

Разбор по частям:

| Часть | Что значит |
|---|---|
| `async def` | Функция асинхронная — внутри будет `await` |
| `test_register` | Имя **обязано** начинаться с `test_` — иначе pytest не найдёт эту функцию и просто пропустит её |
| `client: AsyncClient` | Параметр функции. Pytest видит имя `client`, ищет **фикстуру** с таким именем в `conftest.py`, вызывает её и передаёт результат сюда |

**Как работает подстановка фикстуры:** pytest не запускает тест просто так. Сначала он смотрит на
список параметров функции. Каждое такое имя — это запрос "дай мне то, что готовит фикстура с этим
именем". У тебя в `conftest.py` есть:

```python
@pytest.fixture
async def client(test_db):
    ...
    yield ac
```

Значит когда pytest видит `client: AsyncClient` в параметрах теста — он **сначала** выполнит эту
фикстуру, дождётся `yield ac`, и передаст `ac` (готовый HTTP-клиент) в тест как `client`.

---

```python
    response = await client.post("/users/registration/", json={
        "first_name": "Raul",
        "last_name": "Aitbayev",
        "email": "raul@test.com",
        "password": "12345678"
    })
```

- `client.post(...)` — отправляет POST-запрос на путь `/users/registration/`. Это тот же самый
  путь, который обрабатывает твой роутер `@router.post("/registration/")` в `routers/users.py`.
- `await` — обязателен, потому что `.post()` — асинхронная операция: клиент "ждёт" пока сервер
  (в данном случае — твой FastAPI-код, вызванный напрямую в памяти) обработает запрос и вернёт ответ.
- `json={...}` — тело запроса. Так как `AsyncClient` работает как настоящий HTTP-клиент, эти данные
  автоматически сериализуются в JSON и отправляются с заголовком `Content-Type: application/json`.
  Это ровно то, что ожидает твоя схема `UserCreate` в Pydantic.
- `response` — переменная, в которую сохраняется **весь HTTP-ответ**: статус-код, заголовки, тело.
  Не просто данные — именно объект ответа, у которого есть свои методы и атрибуты.

---

```python
    assert response.status_code == 200
```

- `response.status_code` — числовой HTTP-код, который вернул сервер (200 — успех, 404 — не найдено,
  422 — ошибка валидации, 500 — ошибка сервера).
- `assert условие` — если условие `True`, ничего не происходит, тест продолжается. Если `False` —
  тест **немедленно падает** с `AssertionError`, и pytest покажет, что именно не совпало.

Смысл строки: "Мы утверждаем, что после успешной регистрации сервер обязан вернуть код 200.
Если это не так — что-то сломано, и мы хотим об этом узнать."

---

```python
    assert response.json()["first_name"] == "Raul"
```

- `response.json()` — парсит тело ответа (это строка в формате JSON) и превращает его в обычный
  Python-словарь. Например: `{"first_name": "Raul", "last_name": "Aitbayev", "id": 1}`.
- `["first_name"]` — берём значение по ключу из этого словаря.
- Сравниваем с ожидаемым "Raul".

**Зачем это нужно, если уже проверили статус 200?** Статус 200 говорит только "запрос был успешным
технически". Но он не гарантирует, что сервер вернул *правильные* данные. Этот `assert` — вторая,
более глубокая проверка: "сервер не просто ответил успехом, он ещё и создал пользователя с теми
данными, что мы отправили, и вернул их обратно."

## Тест 2 — Логин

```python
async def test_login(client: AsyncClient):
```

Структура точно такая же — асинхронная функция, получает готовый `client` через фикстуру.
Pytest не переиспользует `client` из первого теста — для **каждого теста** фикстура вызывается
заново (если не указано другое через `scope`).

---

```python
    # Сначала нужно зарегистрировать пользователя
    await client.post("/users/registration/", json={
        "first_name": "Raul",
        "last_name": "Aitbayev",
        "email": "raul2@test.com",
        "password": "12345678"
    })
```

**Почему этот шаг вообще есть в тесте про логин?**

Чтобы протестировать вход в систему, в базе данных должен **уже существовать** пользователь с
такими email и паролем. Тестовая база пустая перед каждым запуском (создаётся и удаляется через
фикстуру `setup_database`), поэтому сначала нужно самому создать этого пользователя — точно так же,
как в `test_register`.

**Почему нет `response = ...` и нет `assert` после этого запроса?**

Потому что здесь регистрация — не то, что мы тестируем в этом конкретном тесте. Она нужна только
как **подготовительный шаг**. Нам всё равно, что конкретно вернул сервер на регистрацию — важен
только факт, что пользователь теперь существует в базе.

**Почему email другой — `raul2@test.com`, а не `raul@test.com`?**

Потому что тестовая БД в рамках одной тестовой сессии — общая. Если бы использовал тот же email,
что и в `test_register`, то при попытке зарегистрировать его второй раз сервер вернул бы ошибку
"email уже занят" (это ты сам реализовал в `services/auth.py`). Разные email в разных тестах —
способ избежать таких конфликтов.

---

```python
    response = await client.post("/users/login/", data={
        "username": "raul2@test.com",
        "password": "12345678"
    })
```

Здесь два важных отличия от запроса регистрации:

### 1. `data=` вместо `json=`

Роут `/users/login/` в твоём коде выглядит так:

```python
@router.post("/login/", response_model=Token)
async def login(form_data: OAuth2PasswordRequestForm = Depends(), ...):
```

`OAuth2PasswordRequestForm` — это специальный класс из FastAPI, который реализует стандарт OAuth2.
Он ожидает данные в формате `application/x-www-form-urlencoded` (как обычная HTML-форма
`<form>...</form>`), а **не** JSON. Поэтому в тесте нужно отправить именно `data=`, а не `json=` —
`httpx` под капотом сам выставит правильный `Content-Type` заголовок.

Если бы отправил через `json=` — FastAPI не смог бы правильно распарсить тело запроса, и получил бы
ошибку 422 (Unprocessable Entity).

### 2. `"username"`, а не `"email"`

Это тоже требование стандарта OAuth2 — поле **всегда** называется `username`, даже если по факту
ты используешь email для входа. Это не связано с твоей бизнес-логикой, это просто фиксированное имя
поля, которое ожидает `OAuth2PasswordRequestForm`. Внутри своего кода (`services/auth.py`) ты сам
решаешь, что делать с этим значением — искать пользователя по email, по логину, как угодно.

---

```python
    assert response.status_code == 200
```

То же самое, что и в первом тесте — проверяем, что вход прошёл успешно на уровне HTTP-статуса.

---

```python
    assert "access_token" in response.json()
```

Здесь другой тип проверки, чем в `test_register`. Разберём разницу:

```python
# в test_register — сравнение с конкретным значением
assert response.json()["first_name"] == "Raul"

# в test_login — проверка наличия ключа
assert "access_token" in response.json()
```

**Почему нельзя сравнить с конкретным значением?** Потому что JWT-токен — это подписанная строка,
которая включает в себя время истечения (`exp`) и генерируется заново при каждом запросе. Он **никогда
не будет одинаковым** между запусками теста. Поэтому бессмысленно писать что-то вроде:

```python
assert response.json()["access_token"] == "eyJhbGci..."  # так никогда не будет работать
```

Вместо этого логичная проверка — "токен *существует* в ответе", то есть ключ `access_token`
присутствует в словаре. Оператор `in` для словаря в Python проверяет именно наличие ключа:

```python
d = {"access_token": "abc123", "token_type": "bearer"}
"access_token" in d  # True
"refresh_token" in d  # False
```

## Итог: что именно проверяют эти два теста

| Тест | Что имитирует | Что проверяет |
|---|---|---|
| `test_register` | Новый человек заполняет форму регистрации | Сервер создаёт пользователя и возвращает 200 + правильные данные |
| `test_login` | Уже существующий человек вводит email и пароль | Сервер аутентифицирует его и выдаёт JWT-токен |

Оба теста работают по одной и той же схеме:

```
1. Собрать данные, которые отправил бы реальный пользователь
2. Отправить запрос через client (как будто это настоящий HTTP-запрос)
3. Получить response — реальный ответ от твоего FastAPI-приложения
4. Проверить через assert, что ответ соответствует ожиданиям
```

## Частые вопросы

**Почему `client` не нужно импортировать явно?**
Потому что это не импорт, а фикстура. Pytest сопоставляет имя параметра функции с именем фикстуры
автоматически — ищет по всем `conftest.py` в проекте.

**Что будет, если `assert` не сработает?**
Тест помечается как `FAILED`, pytest покажет какая именно строка не прошла и какие были значения
слева и справа от `==` (или что не нашлось в случае `in`).

**Нужно ли писать `try/except` в тестах?**
Обычно нет. Если код внутри теста выбросит исключение (например, `KeyError`, если в ответе нет
нужного поля) — pytest сам покажет это как ошибку теста. Не нужно ловить исключения вручную,
это как раз то, что должно "падать" и сигнализировать о проблеме.
